### 반려 사유 (§3 UX 재설계의 전제)

> **"같은 방 N개"가 가장 흔한 케이스인데, 슬롯 바가 동일한 카드를 N개 늘어놓아 화면을 차지하고 절차만 늘렸다.**
> **기존 요금표에서 Select 한 번으로 끝나던 흐름이 더 낫다."** (현업 판정)

**교훈**: 분리 예약은 **소수의 예외 케이스**다. 다수 케이스(동일 상품 N실)의 사용성을 조금이라도 해치는 설계는 채택 불가.
→ 재설계 원칙: **기본 흐름은 지금 그대로 두고, 필요할 때만 분리 옵션이 드러날 것**(§3.3 대안 참조).
> **상위 문서**: [기획서 ② §4.2(b)](spec-b-dotbiz-enhancement.md) · 개발 명세 [P-2](../handoff/v1/dev-spec-en.md)
> **한 줄**: 방 2개를 잡을 때 **트윈 1 + 더블 1**처럼 서로 다른 룸타입·요금제를 **한 번의 예약**으로 담는다.

## 1. 배경 — 현재의 구조적 한계

### 1.1 지금 되는 것 / 안 되는 것

| 구분 | 현재 (실사이트 = 클론 동일) |
|------|------------------------------|
| 검색 | Rooms 2 + Room 1(ADT 2)·Room 2(ADT 2) — **인원 구성은 룸별로 입력 가능** (확인됨: S2·S6 가능) |
| 요금 표시 | 요금제 1행 = **전 객실 합계**(Billing Sum). 예: 1룸 8,884 → 2룸 17,768 |
| 선택 | 요금제 **하나만** 선택 → **2개 방 모두 같은 상품**으로 확정 |
| **불가** | **룸마다 다른 상품** — 룸타입(트윈1+더블1)·요금제·식사·취소정책·등급 혼합 (S1·S3·S4·S5) |
| 우회 | 예약을 **2건 따로 생성** → 코드 2개·투숙객 분리·취소 각각 관리·OP 작업 2배 |
| 투숙객 매핑 | 예약 모달 Travelers 행이 전부 `Room 1` 고정 — 룸별 배정 개념 없음 |

### 1.2 왜 문제인가 (OP 관점)

- 고객 요청의 상당수가 **혼합 구성**("부부는 더블, 아이들은 트윈")인데 시스템이 이를 한 건으로 못 받는다.
- 예약 2건으로 쪼개면: 바우처 2장·인보이스 2건·취소 마감 2개를 따로 챙겨야 하고, 같은 일행인지 시스템이 모른다.
- 실수 지점: 한쪽만 취소, 한쪽만 결제 누락, 투숙객 이름 잘못 배치.

### 2.1 판정에서 도출된 문제 정의 (확정)

> **인원 구성(ADT/CHD)은 이미 룸별로 다르게 입력되지만(S2·S6 가능), 상품(룸타입·요금제·식사·취소정책·등급)은 전 객실 동일이 강제된다(S1·S3·S4·S5 불가).**
> → 분리 예약의 본질은 **"룸 슬롯마다 다른 상품을 담게 하는 것"** 하나다.

- **Phase 1 범위**: S1(+S3·S4·S5 자동 포함) — 하나의 메커니즘으로 4개 시나리오가 동시 해결.
- **제외 확정**: S2·S6(이미 가능) · S7·S10·S11(불필요) · S8(2번 예약으로 해결) · **S9(정산 정책상 진행불가)**.
- **S9 제외의 의미**: **부분 취소를 지원하지 않는다** → 취소는 **예약 전체 단위** → §5 예약 단위 설계가 단순해진다.

## 3. UX — 1차안 반려, 재설계 필요

> ⚠️ **§3.1~3.3은 반려된 1차안**(룸 슬롯 바)이다. 기록 목적으로 남긴다 — 재설계는 **§3.4**부터.

### 3.1 흐름 (반려됨)

```
① 검색: Rooms 2 · Room 1(ADT 2) · Room 2(ADT 2)   ← 기존과 동일
② 룸리스트: 요금표 각 행에 [Room 1에 담기] [Room 2에 담기]
③ 상단 슬롯 바:  Room 1 → 트윈 논리펀더블 ¥8,591 ✕
                 Room 2 → 더블 조식포함  ¥12,937 ✕     합계 ¥21,528
④ 두 슬롯이 모두 차면 [Create Booking] 활성화
⑤ 예약 모달: Travelers 테이블이 Room 1(트윈)/Room 2(더블)로 그룹핑되어 배정
```

### 3.2 UI 규칙 (반려됨)

| 항목 | 규칙 |
|------|------|
| 요금 표시 | **1실 단가**로 표기(현재는 전 객실 합계). 슬롯에 담긴 것들의 합이 총액 — 혼합 구성이면 단순 배수가 성립하지 않으므로 **단가 노출이 전제**다 (§4.1) |
| 담기 버튼 | 해당 요금제가 **그 슬롯의 인원 구성을 수용할 때만** 활성 (예: ADT2+CHD1 슬롯에 2인 정원 룸 → 비활성 + 사유 툴팁) |
| 슬롯 바 | 상단 고정. 각 슬롯: 룸타입·요금제·식사·취소정책·단가·✕(비우기) |
| 교체 | 이미 담긴 슬롯에 다른 요금제를 담으면 **교체**(확인 없이 대체, ✕로 비우기 가능) |
| 동일 구성 빠른 담기 | 모든 슬롯이 같은 인원 구성이면 [전체 슬롯에 담기] 제공 → **기존 사용성 유지**(같은 방 2개 예약이 느려지면 안 됨) |
| 미완성 상태 | 슬롯이 비어 있으면 Create 비활성 + "Room 2를 선택해 주세요" 안내 |
| Rooms 변경 시 | 슬롯 수 증감 — 기존 담긴 내용은 유지, 초과분은 제거 확인 |

### 3.3 예약 모달 변경 (반려안 기준 — 재설계에서도 룸별 표기 자체는 필요)

- Travelers 테이블: 현재 전 행 `Room 1` 고정 → **슬롯별 그룹 헤더**(`Room 1 · Moderate Twin Non Smoking`)로 렌더, 각 룸의 정원만큼 행 생성.
- 아동 행은 **아동이 배정된 슬롯에만** 표시(기존 아동 생년월일 검증 F-5 규칙 유지).
- Billing Rate: 룸별 소계 + 합계.
- Booking Detail: `Room Type` 단일 값 → **룸별 목록**으로 확장.

### 3.4 재설계 대안 — 다수 케이스를 건드리지 않는 방식

**설계 제약(반려에서 도출)**:
1. **기본 화면·흐름은 지금 그대로**. 같은 방 N개 예약은 요금표에서 **Select 한 번**으로 끝나야 한다.
2. 분리 예약 UI는 **평소에 보이지 않아야** 한다 — 필요할 때만 드러난다.
3. 요금표의 금액 의미(**전 객실 합계**)를 바꾸지 않는다 — 1차안은 이걸 1실 단가로 바꿔 다수 케이스의 인지 부담을 키웠다.

**대안 후보**:

| 안 | 방식 | 장점 | 단점 |
|----|------|------|------|
| **가. 예약 후 룸 교체** | 기존대로 Select(전 객실 동일) → **예약서(모달)에서 특정 룸만 [룸 변경]** → 그 룸에 대한 요금 목록에서 교체 | 기본 흐름 **완전 무변경** · 분리는 예외 동선으로 분리 | 예약서에서 요금을 다시 고르는 흐름이 낯설 수 있음 |
| **나. Select 시 룸 지정 (드롭다운)** | Select 버튼을 **[Select ▾]** 로 — 기본 클릭 = 전 객실(현행), 화살표 = "Room 2에만 적용" | 버튼 1개만 바뀜 · 화면 추가 없음 | 숨은 기능이라 발견성 낮음 |
| **다. 분리 모드 토글** | 요금표 상단에 **[룸별로 다르게 선택]** 스위치 — 켤 때만 룸 지정 UI 노출 | 명시적 · 기본 화면 무변경 | 토글을 켜야 함(1스텝) |

→ **권고: 가안(예약서에서 룸 교체)**. 실제 업무 순서("일단 잡고, 한 방만 바꿔줘")와도 맞고 기본 흐름을 전혀 건드리지 않는다.
→ **결정 필요**: 가/나/다 중 선택 (또는 기능 자체 보류).

## 4. 데이터·계약 영향 (개발 검토 필요)

### 4.1 요금 단가 노출 — **선행 과제**

현재 요금 의미론은 `1박 단가 × 박수 × 객실수 = Billing Sum`(전 객실 합계)이다.
혼합 구성에서는 **룸별 단가**가 있어야 합계를 계산할 수 있다 →
검색 응답에 **1실 기준 단가**를 제공하고, 화면은 슬롯 단가의 합으로 총액을 만든다.

**수량 종속 요금(객실 수에 따라 실당 단가가 달라지는 요금)** — 자사 약관에 이미 사례가 있다:

> **Terms & Condition 제4조 ③**: *"동일 숙소·동일 일자에 **4실 이상** 예약 시 **그룹 예약**으로 간주되어, 도매 요금 대신 **별도 요금·취소 규정**이 적용될 수 있다"* (적용 시 제8조에 따라 고객에게 통지 → 유지 여부 확인 → 취소 시 제12조)

→ **3실까지는 실당 단가 합산이 성립**하지만, **4실 이상은 그룹 요금이 적용될 수 있어 단순 합산이 성립하지 않는다**.
→ **설계 반영(제안)**: 슬롯 4개 이상이면 합계 옆에 **"4실 이상 — 그룹 예약으로 별도 요금·취소 규정이 적용될 수 있습니다(약관 제4조 ③)"** 경고를 노출하고, 확정 요금은 담당자 확인 후 안내. **확인 필요**: 이 외에 공급사 단위의 볼륨 할인(예: 2실 동시 예약 시 할인)이 실제로 존재하는지.

### 4.2 인원 구성 그룹 (Occupancy Group)

- 슬롯별 인원(ADT/CHD/나이)이 다르면 **유효한 요금제 목록도 달라진다**.
- 요청: 검색 시 **인원 구성별 요금 목록**을 반환(동일 구성은 1개 그룹으로 묶어 1회 조회).
- 프로토타입은 현재 전 슬롯 공통 목록만 생성 → 그룹별 조회로 확장 필요.

### 4.3 재고 (Availability) — **조회 기준 변경 필요**

**판정(2026-07-17)**: *"재고가 있으니 검색이 되는 것"* — 재고 없는 상품은 애초에 검색 결과에 나오지 않는다. 즉 **표시된 요금 = 재고 있음**.

다만 분리 예약에서는 **조회 기준**이 달라져야 한다:

| | 기존(전 객실 동일 상품) | 분리 예약 |
|---|---|---|
| 조회 기준 | Rooms 2 → **"그 타입 2실 이상 가능"한 것만** 표시 | 슬롯은 **1실 단위** → **"1실 이상 가능"한 것**을 표시해야 함 |
| 문제 | — | 트윈이 **1실만 남은 경우** 기존 기준으로는 목록에서 빠져 **트윈1+더블1 혼합이 불가능**해진다 |

→ **설계(제안)**: 요금 조회는 **1실 기준**으로 하고(프로토타입 구현 완료), 같은 타입을 여러 슬롯에 담을 때만 수량 확인이 필요하다.
→ **확인 필요**: 같은 타입을 N실 담을 때 **잔여 수량 초과를 막을 방법** — ⑴ 요금제별 잔여 수량 제공(정확) ⑵ 미제공 시, 담기는 허용하고 **예약 확정 시점에 실패 안내**(UX 열위). 현 모델에는 상태(available/on_request/unavailable)만 있고 수량이 없다.

### 4.4 취소정책·마감의 룸별 상이 — **확정**

- 슬롯마다 취소정책이 다를 수 있음(S4) → 예약 1건에 **룸별 정책**이 공존.
- **표기 규칙 확정(2026-07-17)**: `Client Cancel DL` 단일 컬럼은 **가장 이른 마감**을 표시하고, 상세에서 룸별로 표기한다.
- **정교화(구현 반영)**: **환불불가 룸이 하나라도 포함되면 예약 전체가 환불불가**로 표기한다 — 취소가 예약 전체 단위(부분 취소 미지원)이므로, 무료취소 룸의 마감을 대표값으로 쓰면 "전체 무료취소 가능"으로 오독될 수 있다.
