# 기능 기획 — 분리 예약 (Split-Room Booking)

> **작성일**: 2026-07-17 · **트랙**: B (닷비즈 내부기능 고도화) · **상태**: 기획 초안 — 8월 확정 대상
> **상위 문서**: [기획서 ② §4.2(b)](spec-b-dotbiz-enhancement.md) · 개발 명세 [P-2](../handoff/v1/dev-spec-en.md)
> **한 줄**: 방 2개를 잡을 때 **트윈 1 + 더블 1**처럼 서로 다른 룸타입·요금제를 **한 번의 예약**으로 담는다.

---

## 1. 배경 — 현재의 구조적 한계

### 1.1 지금 되는 것 / 안 되는 것

| 구분 | 현재 (실사이트 = 클론 동일) |
|------|------------------------------|
| 검색 | Rooms 2 + Room 1(ADT 2)·Room 2(ADT 2) **인원 구성은 룸별로 입력 가능** |
| 요금 표시 | 요금제 1행 = **전 객실 합계**(Billing Sum). 예: 1룸 8,884 → 2룸 17,768 |
| 선택 | 요금제 **하나만** 선택 → **2개 방 모두 같은 룸타입**으로 확정 |
| **불가** | **트윈 1 + 더블 1** (서로 다른 룸타입/요금제 혼합) |
| 우회 | 예약을 **2건 따로 생성** → 코드 2개·투숙객 분리·취소 각각 관리·OP 작업 2배 |
| 투숙객 매핑 | 예약 모달 Travelers 행이 전부 `Room 1` 고정 — 룸별 배정 개념 없음 |

### 1.2 왜 문제인가 (OP 관점)

- 고객 요청의 상당수가 **혼합 구성**("부부는 더블, 아이들은 트윈")인데 시스템이 이를 한 건으로 못 받는다.
- 예약 2건으로 쪼개면: 바우처 2장·인보이스 2건·취소 마감 2개를 따로 챙겨야 하고, 같은 일행인지 시스템이 모른다.
- 실수 지점: 한쪽만 취소, 한쪽만 결제 누락, 투숙객 이름 잘못 배치.

## 2. 고객(OP) 시나리오 — 현장 니즈 파악

> 우선순위: **P0** = 1차 범위 확정 후보 · **P1** = 2차 검토 · **P2** = 별도 과제

| # | 시나리오 | 상황 예시 | 현재 | 우선순위 |
|---|----------|-----------|------|----------|
| S1 | **룸타입 혼합** (핵심 요청) | 트윈 1 + 더블 1 — 부부+친구, 부모+자녀 | 불가 | **P0** |
| S2 | **인원 구성 혼합** | Room 1 ADT 2 + Room 2 ADT 1(싱글 유즈) — 홀수 인원 일행 | 입력은 되나 룸타입은 동일 강제 | **P0** |
| S3 | **요금제(식사) 혼합** | 한 방은 조식 포함, 다른 방은 룸온리 — 예산 차이 | 불가 | **P0** |
| S4 | **취소정책 혼합** | 확정 인원은 논리펀더블(저가), 미정 인원은 무료취소 | 불가 | **P0** |
| S5 | **룸 등급 혼합** | 임원 디럭스 + 직원 스탠다드 — 기업 출장 서열 | 불가 | **P0** |
| S6 | **가족 분리** | Room 1 성인 2+아동 1(패밀리룸), Room 2 성인 2 | 아동 나이 입력은 되나 룸타입 혼합 불가 | **P0** |
| S7 | **흡연/금연 혼합** | 흡연 1 + 금연 1 | 불가 | **P0** |
| S8 | **투숙 기간 상이** | A는 3박, B는 2박(늦게 합류/먼저 출발) | 불가 — 예약 2건 필수 | **P1** |
| S9 | **개별 취소 통제** | 트윈 5개 중 2개만 취소 가능해야 함 (단체 인원 변동) | 예약을 5건으로 쪼개야 통제 가능 | **P1** |
| S10 | **국적별 요금 혼합** | 방마다 투숙객 국적이 달라 판매가/가능여부 상이 | 미지원 (ELLIS 지원 여부 미확인 — R2) | **P2** |
| S11 | **호텔 상이** | 일행이 다른 호텔에 분산 | 예약 2건이 정상 — 분리 예약 대상 아님 | 대상 외 |

**해석**: S1~S7은 전부 **"같은 호텔·같은 기간, 방마다 다른 상품"** 이라는 하나의 구조로 묶인다 → **분리 예약 1차 범위**로 함께 해결 가능.
S8(기간 상이)과 S9(개별 취소)는 예약 단위 설계(§5)에 따라 가능 여부가 갈리므로 2차.

## 3. 기획안 — 룸 슬롯 방식 (Room Slot / 장바구니)

### 3.1 흐름

```
① 검색: Rooms 2 · Room 1(ADT 2) · Room 2(ADT 2)   ← 기존과 동일
② 룸리스트: 요금표 각 행에 [Room 1에 담기] [Room 2에 담기]
③ 상단 슬롯 바:  Room 1 → 트윈 논리펀더블 ¥8,591 ✕
                 Room 2 → 더블 조식포함  ¥12,937 ✕     합계 ¥21,528
④ 두 슬롯이 모두 차면 [Create Booking] 활성화
⑤ 예약 모달: Travelers 테이블이 Room 1(트윈)/Room 2(더블)로 그룹핑되어 배정
```

### 3.2 UI 규칙

| 항목 | 규칙 |
|------|------|
| 요금 표시 | **1실 단가**로 표기(현재는 전 객실 합계). 슬롯에 담긴 것들의 합이 총액 — 혼합 구성이면 단순 배수가 성립하지 않으므로 **단가 노출이 전제**다 (§4.1) |
| 담기 버튼 | 해당 요금제가 **그 슬롯의 인원 구성을 수용할 때만** 활성 (예: ADT2+CHD1 슬롯에 2인 정원 룸 → 비활성 + 사유 툴팁) |
| 슬롯 바 | 상단 고정. 각 슬롯: 룸타입·요금제·식사·취소정책·단가·✕(비우기) |
| 교체 | 이미 담긴 슬롯에 다른 요금제를 담으면 **교체**(확인 없이 대체, ✕로 비우기 가능) |
| 동일 구성 빠른 담기 | 모든 슬롯이 같은 인원 구성이면 [전체 슬롯에 담기] 제공 → **기존 사용성 유지**(같은 방 2개 예약이 느려지면 안 됨) |
| 미완성 상태 | 슬롯이 비어 있으면 Create 비활성 + "Room 2를 선택해 주세요" 안내 |
| Rooms 변경 시 | 슬롯 수 증감 — 기존 담긴 내용은 유지, 초과분은 제거 확인 |

### 3.3 예약 모달 변경

- Travelers 테이블: 현재 전 행 `Room 1` 고정 → **슬롯별 그룹 헤더**(`Room 1 · Moderate Twin Non Smoking`)로 렌더, 각 룸의 정원만큼 행 생성.
- 아동 행은 **아동이 배정된 슬롯에만** 표시(기존 아동 생년월일 검증 F-5 규칙 유지).
- Billing Rate: 룸별 소계 + 합계.
- Booking Detail: `Room Type` 단일 값 → **룸별 목록**으로 확장.

## 4. 데이터·계약 영향 (개발 검토 필요)

### 4.1 요금 단가 노출 — **선행 과제**

현재 요금 의미론은 `1박 단가 × 박수 × 객실수 = Billing Sum`(전 객실 합계)이다.
혼합 구성에서는 **룸별 단가**가 있어야 합계를 계산할 수 있다 →
검색 응답에 **1실 기준 단가**(및 필요 시 객실수 적용 총액)를 함께 제공해야 한다.
※ 공급사가 "2실 동시 예약 시 할인" 같은 **수량 종속 요금**을 주는 경우 단순 합산이 틀어질 수 있음 — **확인 필요**.

### 4.2 인원 구성 그룹 (Occupancy Group)

- 슬롯별 인원(ADT/CHD/나이)이 다르면 **유효한 요금제 목록도 달라진다**.
- 요청: 검색 시 **인원 구성별 요금 목록**을 반환(동일 구성은 1개 그룹으로 묶어 1회 조회).
- 프로토타입은 현재 전 슬롯 공통 목록만 생성 → 그룹별 조회로 확장 필요.

### 4.3 재고 수량 (Availability Count)

- "트윈 1개만 남음"인데 2개를 담을 수 없어야 한다 → 요금제별 **잔여 수량** 필요.
- 현재 모델에 재고 수량 개념 없음(available/on_request/unavailable 상태만).

### 4.4 취소정책·마감의 룸별 상이

- 슬롯마다 취소정책이 다를 수 있음(S4) → 예약 1건에 **룸별 정책**이 공존.
- Bookings 그리드의 `Client Cancel DL` 단일 컬럼 표기 규칙 필요: **가장 이른 마감** 표시 + 상세에서 룸별 표기 (제안).

## 5. 예약 단위 — **결정 필요 (블로킹)**

같은 일행의 혼합 2룸을 무엇으로 발번하는가?

| 옵션 | 구조 | 장점 | 단점 |
|------|------|------|------|
| **A. 1 예약 = N 룸** | ELLIS 코드 1건 + `rooms[]` (룸별 타입·정책·투숙객) | OP가 코드 하나로 관리 · 바우처/인보이스 1건 · 일행 관계 명확 | **부분 취소(S9)** 지원 필요 · 룸별 정책 혼재로 그리드/정산 표기 복잡 |
| **B. N 예약 + 그룹 ID** | 룸당 코드 1건, `group_id`로 묶음 | 룸별 취소·결제 독립(S9 자연 해결) · 기존 예약 모델 그대로 | 목록에 2행 → 일행 인지 위해 그룹 표기 UI 필요 · 바우처 2장 |
| **C. 하이브리드** | 발번은 B, **화면에서는 묶음으로 표시** | 백엔드 단순 + OP 경험은 1건처럼 | 그룹 UI 개발 필요 |

- **ELLIS/실사이트 확인 필요**: 실제 시스템이 예약 1건에 복수 룸을 담을 수 있는지, 부분 취소가 가능한지.
- 확인 전까지 프로토타입은 **C(하이브리드)** 로 시연하는 것을 제안 — 어느 쪽으로 확정돼도 화면 흐름은 유지된다.

## 6. 후속 영향

| 영역 | 영향 |
|------|------|
| Bookings 그리드 | `Room Type / Count` 표기: 혼합 시 `Twin[1] + Double[1]` (제안) · 그룹 예약 표시(옵션 B/C) |
| 예약 상세 | 룸별 섹션(타입·요금·정책·투숙객) |
| Voucher / Invoice | 1건에 룸별 명세 (옵션 A/C) |
| 취소 | 부분 취소 지원 여부(§5) |
| AI 요금 검색 (트랙 A) | 파서는 이미 **"더블+트윈 각각 1개씩"** 을 인식(`room_types`, `rooms`) → 인식 결과를 **슬롯 자동 담기**로 연결 가능 |
| 룸타입 솔팅 (P-1) | 슬롯에 담을 후보를 좁히는 필터로 함께 동작 |

## 7. 단계 구분 (제안)

| 단계 | 범위 |
|------|------|
| **Phase 1** | S1~S7 — 같은 호텔·같은 기간, 슬롯별 룸타입/요금제/인원 혼합. 단가 노출·인원 구성 그룹·슬롯 UI·투숙객 룸 배정 |
| **Phase 2** | S8(기간 상이) · S9(부분 취소) — 예약 단위 확정(§5) 이후 |
| **대상 외** | S11(호텔 상이) — 예약 분리가 정상 · S10(국적별) — ELLIS R2 확인 후 별도 |

## 8. 미결정 사항 (기획 확정 전 회신 필요)

1. **예약 단위** (§5): 1코드+N룸 / N코드+그룹 / 하이브리드 — **ELLIS·실사이트 확인 필요**
2. **부분 취소** 가능 여부 (S9) — 예약 단위와 연동
3. **수량 종속 요금** 존재 여부 (§4.1) — 2실 동시 예약 할인 등
4. **재고 수량** 제공 가능 여부 (§4.3)
5. **`Client Cancel DL` 단일 컬럼 표기 규칙** (§4.4) — 가장 이른 마감 표시 제안
6. Phase 1 범위에 S8(기간 상이)을 포함할지 — 현장 빈도 확인 필요
