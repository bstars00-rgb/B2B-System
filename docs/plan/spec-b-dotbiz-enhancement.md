## 1. 배경과 목적

### 1.1 문제 인식

1. **물량의 API 의존**: 상당수 거래 물량이 외부 API 연동(어그리게이터/채널 경유)에 의존한다.
   API 물량은 ① 마진 구조가 상대적으로 불리하고 ② 고객사와의 직접 관계가 형성되지 않으며 ③ 노출·프로모션 통제력이 없다.
2. **닷비즈 직접 사용의 불편**: 닷비즈(Ohmy Partners)를 직접 쓰는 셀러 OP들은 검색·예약 과정의
   자잘한 불편(느낌상 멈춘 검색, 과거 날짜 실수, 룸 구성 변경 불가, 견적 재입력 등)으로 이탈하거나 API로 회귀한다.

### 1.2 목적

> **닷비즈를 사용하면서 불편한 점을 하나씩 개선해 고객 친화적인 플랫폼을 만들고,
> 기존에 API 연동에 의지했던 물량을 닷비즈로 옮겨 고객사 확보와 물량 확보를 동시에 달성한다.**

| 축 | 내용 |
|----|------|
| 고객 친화 | 현장 OP가 매일 쓰는 화면의 불편을 백로그로 수집해 순차 제거 |
| 고객사 확보 | 직접 가입·온보딩 장벽 완화(다국어·매뉴얼·복지 혜택) → 신규 셀러 유입 |
| 물량 전환 | API 경유 물량을 닷비즈 직접 예약으로 이관 → 마진·데이터·관계를 회사가 확보 |

### 1.3 기획서 ①과의 관계

AI 요금 검색(트랙 A)은 **차별화 신기능**, 본 기획(트랙 B)은 **기본기 강화**다.
트랙 B는 ELLIS API·LLM 없이도 배포 가능하며, 트랙 A 파일럿의 성공 전제(쓸만한 포털)를 만든다.

## 3. 운영 방식 — 불편 개선 백로그

**프로세스**: 현장 지목(스크린샷/한 줄 요청) → 실사이트 대조 → 프로토타입 구현·검증 → 당일 배포 → 라이브 확인.
한 번에 크게 고치지 않고, **하나의 불편 = 하나의 커밋 = 당일 반영**을 원칙으로 한다.

**고도화 표기 (2026-07-17 도입)**: 닷비즈 원본과 다른(고도화된) 지점에는 화면에 보라색 **`UP` 배지**를 표기한다 —
마우스 오버 시 어떤 고도화인지 툴팁으로 설명되어, 기획자가 데모만 보고도 원본과의 차이를 한눈에 파악할 수 있다.
헤더의 **✨ 버튼**으로 전체 표기를 켜고 끌 수 있다(끄면 순수 클론 화면 — 설정은 localStorage 유지).
실사이트에 원래 있는 기능을 따라 만든 것(패리티)에는 배지를 달지 않는다.

### 4.2 [KR2] 예약생성 OP 편의 기능 (현장 니즈 기획 업그레이드)

현장에서 필요한 기능들을 하나씩 기획해 업그레이드한다. 1차 확정 3건:

#### (a) 룸타입 솔팅 — 트윈/더블 정렬·필터

| 항목 | 내용 |
|------|------|
| 불편 | 룸리스트 요금제가 가격순 나열뿐 — "트윈만 보여줘"가 안 됨. 고객 요청은 대부분 침대 타입 기준 |
| 개선 | 룸리스트 요금 테이블 상단에 **룸타입 칩 필터**(All/Double/Twin/Suite…— 결과에서 자동 추출) + 정렬(가격/룸타입) 추가 |
| 동작 | 칩 선택 시 해당 타입 요금제만 표시 · 복수 선택 가능 · Show more와 독립 동작 |
| 수용 기준 | Twin 선택 시 Twin 계열만 표시 · 0건이면 "해당 타입 없음" 안내 · 필터 상태가 재검색 후에도 유지 |

#### (b) 더블·트윈 분리 예약

| 항목 | 내용 |
|------|------|
| 불편 | 한 고객이 "더블 1 + 트윈 1"을 원할 때 현재는 예약을 **두 번** 생성해야 함 (요금 선택이 단일 플랜) |
| 개선 | 룸리스트에서 요금제를 **장바구니처럼 룸 슬롯에 담기**: Rooms 2 검색 시 각 룸 슬롯에 서로 다른 요금제 배정 → 한 번의 Create로 묶음 예약 |
| 동작 | ① Rooms 2 검색 → 요금제 행에 [Room 1에 담기/Room 2에 담기] → ② 담긴 구성 합산 요금 표시 → ③ 예약 모달 Travelers가 룸·룸타입별로 매핑 → ④ 생성 시 ELLIS 코드 1건 + 룸 상세 2행 (또는 코드 2건 발번 — 실사이트 규칙 확인 필요) |
| 수용 기준 | 더블+트윈 각 1실 묶음 생성 → Bookings 상세에서 룸별 타입·투숙객 확인 가능 · AI 검색의 "더블+트윈 각각 1개씩" 인식과 연결 |
| 확인 필요 | 실사이트/ELLIS의 묶음 예약 단위(코드 1건 vs 룸당 1건) — 실물 스크린샷 필요 |

#### (c) 요금·룸타입 복사 (견적 공유)

| 항목 | 내용 |
|------|------|
| 불편 | OP가 고객에게 견적을 보낼 때 호텔명·기간·룸타입·금액·취소규정을 **손으로 옮겨 적음** — 오타·누락 사고 |
| 개선 | 요금제 행과 예약 상세에 **[복사] 버튼** — 정형 견적 텍스트를 클립보드로 |
| 복사 포맷(초안) | `[Hotel AMANEK Osaka Namba] 2026-07-22~23 (1박) · Standard Double (Non Smoking) · JPY 8,884 · 무료취소 2026-07-19 17:00까지` — 언어팩 연동(고객 언어로 복사) |
| 수용 기준 | 클릭 1번으로 복사 완료 토스트 · 다중 요금제 선택 복사(비교 견적) · 금액은 도구 데이터 그대로(수기 가공 없음) |

#### (d) 현장 백로그 수집 채널

초기에는 본 문서 §3.2 표를 운영 채널로 사용(지목 → 표 추가 → 구현). 분기별로 완료 건수를 KR2 지표로 집계.
