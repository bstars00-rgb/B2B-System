## 2. 데모 확인 방법

1. 데모 접속 → 로그인 → 좌측 **Dashboard**(Seller 첫 항목, `UP` 배지).
2. **Overview** — KPI 4장, 일별 차트, 12개월 TTV, 목적지 파이+표, 베스트셀러 랭킹.
3. **필터를 직접 조작해 볼 것 — 화면만이 아니라 실제로 동작한다.** 기간을 `This Year`로 바꾸면 KPI가 재계산되고, 집계 기준일을 `Booking Date` → `Check-in Date`로 바꾸면 **181건 → 63건**으로 떨어진다. 체크인 대부분이 아직 미래이기 때문. "이번 달 **투숙** 기준으로 얼마나 있지?"는 셀러가 실제로 묻는 질문이고, 두 축의 답이 다르다.
4. **Bookings와 대조해 볼 것**: 좌측 Bookings를 열어 비교하면 수치가 맞아떨어진다. 대시보드는 그 목록을 집계한 것이다.
5. 나머지 탭: Booking Statistics(6개월 확정/취소/이연) · Cancellation(취소율+사유) · Daily · Year-End.
6. **Year-End의 2024~2025는 비어 보인다.** 의도된 것이며 §5에 사유가 있다 — 탭을 판단하기 전에 읽어주기 바란다.

## 3. 만든 것

### 3.1 화면 (5탭)

| 탭 | 구성 |
|----|------|
| **Overview** | KPI 4종(총 예약·총 매출·룸나이트·건당 평균) · 일별 통계 · 12개월 TTV · 목적지 비중(국가/도시) · 베스트셀러 랭킹 |
| **Booking Statistics** | 6개월 확정/취소/이연 추이 |
| **Cancellation** | 취소율 추이 + 사유 분포 |
| **Daily** | 일별 시계열(건수/금액/박수) |
| **Year-End** | 3개년 비교 + YoY 성장률 표 |

PD팀의 **미니멀 원칙을 그대로 유지**했고, 계속 지킬 것을 권한다 — *예약 관련 지표만. AR Aging·Dispute·SLA·Loyalty Tier는 각 전용 화면에.* "지표 하나만 더" 요청이 올 때 거절할 근거가 이 원칙이고, 화면이 읽히는 이유가 그것이다.

### 3.2 가장 중요하다고 보는 지점: 전부 파생

대시보드에 **저장된 숫자가 없다.** 모든 KPI·차트·표가 예약 목록에서 명시적 계산식으로 나온다.

실무적 이유가 있다 — 셀러는 대시보드와 Bookings 목록을 나란히 놓고 본다. 손으로 적은 수치는 데이터가 바뀌는 순간 결함이 된다. 실제로 대조 검증했다: 화면의 *This Month* KPI(43건 / ¥2,662,009 / 189박)가 저장된 예약을 7월로 집계한 값과 일치하고, *This Year*는 181건 / ¥9,393,521 / 685박이다.

부수 효과로 **집계 기준일·기간 셀렉트가 실제로 동작하게 됐다.** 1차 구현에선 화면만이었는데, 데이터가 파생되니 추가 비용 없이 진짜 필터가 됐다.

의도적 예외 하나: **베스트셀러 랭킹은 파생이 아니다.** 플랫폼 전체 랭킹이라 셀러 한 곳의 예약에서 나올 수 없고, 별도 데이터 소스가 필요하다(§6-②).

### 3.3 랭킹에서 클릭 한 번으로 예약

베스트셀러를 본 셀러는 그 호텔을 예약하고 싶어진다. 이제 호텔명을 누르면 **목적지·호텔이 채워지고 달력이 열린 Create Booking**으로 이동한다 — 남은 일은 날짜 선택뿐.

이걸 실제로 만들면서 짚어둘 문제가 드러났다. 랭킹 목데이터는 **102개 도시의 호텔 320개**(두바이·파리·뉴욕)였는데 그중 **우리 예약 가능 인벤토리에 실재하는 건 정확히 1개**였다. "OhMyHotel 베스트셀러"인데 정작 OhMyHotel에서 예약할 수 없는 목록은 그 자체로 모순이고, 클릭해도 갈 곳이 없다. 랭킹을 호텔 마스터에서 만들도록 재구성해 모든 행이 실재하는 예약 가능 호텔로 연결된다.

**운영에서는 이것이 랭킹 API의 요구사항이 된다** — 랭킹에 오른 호텔은 전부 예약 가능해야 하고, 각 행이 호텔 코드를 실어 보내야 검색으로 딥링크할 수 있다.

### 3.4 빈칸에 무엇을 넣었나

전폭 화면에 5컬럼이라 대부분이 여백이었다. 추가한 것:

| 컬럼 | 넣은 이유 |
|------|-----------|
| **MoM** (`▲3` / `▼2` / `NEW`) | 랭킹에서 가장 정보량이 큰 값 — 무엇이 뜨고 무엇이 지는가. API가 전월 순위를 함께 줘야 한다 |
| **Hotel Code** | 셀러가 검색창에 그대로 붙여넣는 6자리 코드 — 행에서 가장 즉시 실행 가능한 값 |
| **Nightly from** | 1박 참고 요금. 고객 예산에 맞는 걸 훑어볼 수 있다. **참고값임을 화면에 명시** — 실제 요금은 날짜·인원·요금제에 따라 다르다 |
| **Chain Brand** | 법인 고객은 체인 선호가 있다. Create Booking의 좌측 필터 차원과도 같다 |

**Property Type 컬럼은 검토 후 제외**했다 — 우리 호텔 114개 중 112개가 그냥 `Hotel`이라 폭만 먹고 정보가 없다. 운영 데이터가 풍부하면 재검토할 만하다.

### 3.5 데모 데이터

대시보드가 의미를 가지려면 데이터가 필요해 프로토타입에 **예약 200건**을 만들었다: 예약일 2026-05-01~오늘(미래 예약 없음), 체크인은 항상 예약일 이후, 취소율 9.5%. 호텔은 우리 호텔 DB(15개 도시·114개 실제 호텔)에서 뽑았고, 200건이 실제로 닿는 곳은 **14개 도시·63개 호텔**이다 — 일본·비즈니스호텔에 가중을 둬 데모 셀러(ATTIC TOURS, 일본계 여행사)의 실제 예약 성향에 맞췄다.

금액은 **지어내지 않았다** — 각 예약 금액을 호텔 DB의 실제 1박 요금(× 박수 × 룸수)에서 산출해 엔화로 환산했다. 그래서 대시보드 매출과 호텔 검색이 같은 요금을 근거로 한다.

## 5. 솔직한 한계 1건

프로토타입 예약이 **2026-05-01**부터라 긴 이력이 필요한 곳은 비어 있다.

| 화면 | 상태 |
|------|------|
| Overview · Cancellation · Daily | 정상 |
| 12개월 TTV | 12개월 중 3개월 |
| Booking Statistics(6개월) | 6개월 중 3개월 |
| **Year-End(3개년)** | **2024·2025 통째로 비어 있음** |

**이력을 지어내 채우지 않았다.** 2024~2025 예약을 만들어내면 이번 라운드가 만들려던 성질 — 대시보드가 실제 예약과 맞는다 — 이 그대로 깨진다. 대신 화면에 사유를 표기했다(Year-End엔 안내 배너).

**운영에선 문제가 아닐 가능성이 높다**: 실제 셀러는 실제 이력이 있다. 프로토타입 데이터의 산물이다. 다만 신규 셀러 계정에는 같은 질문이 남는다 — §6-⑥.

## 7. 운영 적용 전 필요 사항

프로토타입은 `localStorage`를 브라우저에서 집계한다. 운영엔 두 가지가 반드시 바뀌어야 한다([dev-spec §7](dev-spec-en.md#7-backend-requirements-production)).

1. **집계를 서버로.** 3년치 예약을 가진 셀러의 전체 목록을 브라우저로 내릴 수 없다. 파생 계층(`utils/dashboardStats.ts`)은 UI 의존성 없는 순수 모듈 하나로 일부러 분리해 뒀으니 그대로 이식 가능하다.
2. **취소 시 `cancel_reason` 수집.** 없으면 Cancellation 탭은 **데이터 소스 자체가 없다** — 현재 예약 흐름은 사유를 기록하지 않는다. 취소 시 사유 선택 UI와, 사유 없는 과거 취소 건 처리 방침이 필요하다.

그 외: 차트에 Recharts(약 350kB) 사용. 프로토타입엔 수용 가능하나 운영은 대시보드 라우트 code-split 권장.
